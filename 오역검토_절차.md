# SDGG 카드다스 아카이브 — 오역 검토 / 요미가나 확인 절차

> 데이터·이미지 작업은 **`C:\ReddStudios\SDGGCarddas\`** 에서만. `E:\...\2_카드다스업스케일\` 원본 PNG는 **읽기 전용 참조**(고해상도 OCR용)로만 사용.

## 0. 공통 사실
- 배포 데이터: `cards.js` (`window.CARDS = [PRODUCTS...]`). **여기가 1차 수정 대상.** 스크래치패드 `data_*.json`은 원본 백업(동기화 권장).
- **cards.js는 카드 1장 = 1줄** 형식으로 유지(제품/편/시리즈 헤더만 별도 줄, 카드 객체는 `{...}` 한 줄). git diff 가독성을 위함이니 **수정 시 이 형식을 깨지 말 것** — 고유 문자열 치환은 해당 카드 줄만 바꾸므로 안전. 만약 구조를 다시 만들었다면 `scratchpad/reformat.js`(eval 후 카드별 `JSON.stringify` 1줄 직렬화, 재평가 서명 일치 검증)로 재배치.
- 카드 객체 키 순서: `{no,img,jdesc,kdesc,jn,kn,v,jd,kd}`. **`img`는 `.jpg`** (예 `"11200790.jpg"`). 앵커링크 `#11200790` = img에서 확장자 제거.
- 원본 이미지: `E:\...\2_카드다스업스케일\<제품>\<편>\<시리즈>\<img>.png` (cards.js의 `dir` + img, 단 E:는 `.png`).
- 카드 레이아웃(가로 860×590): 상단 띠 = 카드번호+대사, 하단 띠 = 엠블럼 | 이름 | 설명 | 수치. **한자 위에 작은 후리가나**가 붙음. 세로 카드(590×860)는 좌표 다름.
- 파일 쓰기: UTF-8 **BOM 없이** (`New-Object System.Text.UTF8Encoding($false)`). 수정 후 `node`로 `JSON.parse` 검증.

## A. 오역 신고 시트 검토 절차
1. 시트 CSV 받기(공개 권한 필요): `https://docs.google.com/spreadsheets/d/<ID>/export?format=csv`. **바이트로 받아 UTF-8 디코딩**(WebClient.DownloadData → WriteAllBytes), 안 그러면 한글 깨짐.
2. 각 행 = (타임스탬프, 앵커링크, 오역범위, **제안**). 앵커(`#xxxxxxxx`)로 `cards.js`에서 해당 카드(img=앵커+`.jpg`)와 `dir` 조회. (node로 CARDS eval 후 검색)
3. **번역 제안(제안 컬럼) 신뢰도 평가 — 신뢰도가 높으면 카드 원본을 확인하지 않고 바로 적용.**
   - **신뢰도 높음 → 4번(원본 OCR) 생략하고 5번으로 직행.** 아래를 충분히 만족할 때:
     - 명백한 표기 교정(오타·띄어쓰기·철자·일관성)이며 의미를 바꾸지 않음.
     - 정발/공식 건담 고유명사, 또는 아카이브 내 같은 명칭의 정착 표기와 일치.
     - 일본어 원문(jn/jd)·후리가나 현지화 규칙(B 절차)과 논리적으로 합치.
     - 변경 범위가 한정적이고 자체 검증 가능(원문 해석을 바꾸지 않고 한글 표기만 조정 등).
   - **신뢰도 낮음/모호 → 4번대로 원본 확인 후 판단.** 예: 의미를 바꾸는 재작성, 한자/카타카나 판독 자체가 쟁점(OCR 오류 의심), 일본어 원문과 모순, 근거 불명·주관적, 기존 표기와 충돌.
   - 제안이 비어 있으면(선택 필드) 4번대로 진행. *애매하면 항상 원본 확인 쪽으로.*
4. (신뢰도 낮음 또는 제안 없음일 때) 원본 PNG의 **신고 범위 영역을 확대 크롭**해서 Read로 재OCR:
   - 이름/설명 → 하단 띠 `Rectangle(0,476,860,114)` 부근, 4~4.5배 확대.
   - 대사 → 상단 띠 `Rectangle(0,0,860,86)`, 2.5~3배.
   - 한자 판독은 **후리가나(위 작은 글자)** 로 교차검증.
5. 현재값과 대조해 수정(신뢰도 높은 제안은 제안값을, 아니면 OCR 결과를 반영):
   - **캐릭터 한글 이름**: 현지화된(정발/관용) 이름으로. 후리가나가 카타카나 영문명이면 그 발음으로(→ B 절차 참고).
   - 대사/설명: OCR 오류 교정(한자 오독 주의: 裏↔梟, 訳↔獣, 雷↔貴, 嘆↔導 등).
   - 이름/칭호 변경은 **전 시리즈·대사까지 전역 치환**으로 일관 적용(같은 명칭이 다른 카드·대사에도 쓰임).
   - ⚠️ **고유명(캐릭터/유닛/무기) 변경 시 반드시 전역 전파 검증**: 신고된 그 필드(kn 등)만 바꾸면 안 됨. 옛 고유명 코어 문자열을 cards.js 전체(kn/kdesc/kd, **다른 제품/편 포함**)에서 검색해 남은 곳을 모두 새 표기로 치환할 것. 적용 후 옛 표기 잔존 0 확인. (예: メターナル=메터널→메타날을 다른 제품 카드 대사까지, ガブスレイ=가브슬레이→가브스레이를 1편 카드까지.)
6. `cards.js` + 해당 스크래치패드 `data_*.json`에 **고유 문자열 치환**으로 적용(치환 건수 확인). `JSON.parse` 검증.
7. 처리 기록을 프로젝트 루트 `report_processed.json`(GitHub 관리)에 남겨 **재처리 방지**(신뢰도 높아 바로 적용한 건은 result에 그 취지 기록). 시트는 쓰기 권한이 없으면 사용자가 해당 행 삭제/표식.
8. 사용자에게 `git add cards.js && commit && push` 안내.

## B. 요미가나(후리가나) 칭호 확인 절차
1. 대상 후보를 cards.js에서 추출(예: kn이 `~기사` 리터럴, 또는 특정 한자 포함). 같은 JP 칭호 그룹별로 묶고 대표 카드 1장 선정.
2. 대표 카드 이름 영역을 **고배율 크롭**(`Rectangle(106,484,305,104)` ×4.5 등)해서 후리가나 판독.
3. 판정:
   - **카타카나(영문 클래스명)** → 그 발음을 한글로. 예: 鎧騎士=アーマーナイト→아머나이트, 灼熱騎士=バーニングナイト→버닝나이트, 貴騎士=ノーブルナイト→노블나이트, 皇騎士=クラウンナイト→크라운나이트, 邪騎士=エビルナイト→이블나이트, 銀騎士=シルバーナイト→실버나이트, 妖騎士=ファントムナイト→팬텀나이트.
   - **히라가나(한자 음독)** → 한글 번역 유지(예: 近衛騎士=このえきし→근위기사, 龍騎士=りゅうきし→용기사).
   - 'K건담' 같은 약칭은 후리가나(キング)대로 → 킹건담.
   - 'O'(皇)처럼 후리가나가 クラウン이면 대사에서도 황→크라운.
4. 표기 통일: 같은 한자가 제품마다 한글이 다르면 하나로 통일.
5. cards.js의 kn(이름)만 치환(jn 일본어 원문은 인쇄 그대로 유지). `JSON.parse` 검증 후 git 안내.

### ⚠️ 주의 (반드시)
- **같은 한자라도 카드/제품마다 후리가나가 다를 수 있음.** 예: 聖騎士가 주인공 카드(12302610)에선 クラウンナイト, 다른 카드에선 セイント/せい일 수 있음. → 대표 1장 일괄적용은 위험. 다인원 그룹은 카드별 확인 권장.
- **저장된 jn 한자 자체가 OCR 오류**인 경우 다수(原示→啓示, 参→蛮, 操紅→深紅, 神裂→神秘 등). 후리가나/이미지로 실제 한자 확인.
- 카타카나→한글 음역 표기(블랑슈/디바인/리벨 등)는 **사용자 확정** 필요.
- 콤바인/합체 카드 등 일부는 이름 영역 좌표가 달라 아트워크가 잡힘 → 좌표 조정 재크롭.
